### 1.1 需求来源与讨论脉络

本 PRD 不是最终拍板版，而是用于和需求方继续沟通的交流版。里面分为三类标注：

- `已确认`：目前用户已经明确表达过的方向。
- `当前假设`：为了推进 MVP 暂时采用的产品判断，需要后续确认。
- `待确认`：必须向需求方继续问清楚的问题。

#### 1.1.1 已确认方向

- `已确认` 这个工具首先是工作辅助，不是个人资讯阅读产品。
- `已确认` 主要服务对象是公司市场部。
- `已确认` 最终目标不是只看一家公司的官网更新，而是看目标公司所在领域的行业热点和竞品动态。
- `已确认` 老板需要一个 dashboard，而不只是 Markdown 日报或本地脚本输出。
- `已确认` MVP 先拿 ACRO / 百普赛斯作为标本对象验证机制。
- `已确认` 现在已加入 Thermo Fisher Scientific 作为第二家公司样本。
- `已确认` 公司池已进一步扩展默克生命科学、Sartorius 和美天旎，用于验证生物工艺、实验室解决方案与细胞治疗领域的对标监测。
- `已确认` 需要参考 AIhot 的形态：精选流、日报、主题、API / Agent 接入，而不是普通新闻站。

#### 1.1.2 用户原始思路整理

用户最初的理解是：系统可能是每天看一家公司的官网最近有没有更新新闻。但讨论后发现，仅做官网更新监控太窄，且容易带来误推。

当前更准确的方向是：

- 收集目标公司、竞品和相关行业主题的公开动态。
- 不因为一个公司名出现在某条新闻里就立刻推送。
- 需要有筛选、打分、归档和推送分层机制。
- 需要控制成本，不能一开始就依赖大量付费 API 或高频爬取。
- 需要先用低成本公开来源、规则筛选和每日更新验证价值。
- 后续再考虑 AI 摘要、即时提醒和团队化使用。

#### 1.1.3 当前假设

- `当前假设` 市场部每天更需要 3-20 条精选，而不是几百条全量新闻。
- `当前假设` 默认更新频率为工作日每天 1 次，比实时推送更适合 MVP。
- `当前假设` GitHub Pages 静态 dashboard 足够支撑早期演示和需求确认。
- `当前假设` 公开版只放样本公司；如果导入真实公司名单，需要确认是否可以公开。
- `当前假设` 先用规则评分控制噪音，再把 AI 用在高分候选的摘要和业务建议上。

#### 1.1.4 需要带去沟通的问题

- `待确认` 需求方说的“热点新闻平台”到底更偏新闻聚合、竞品监控、市场选题，还是老板经营看板？
- `待确认` 目标公司名单是否可以放在公开 GitHub Pages 上，还是必须部署为公司内部页面？
- `待确认` 市场部实际想看的“竞品动态”包括哪些类型：产品发布、展会、合作、融资、临床 / 监管、内容营销、价格 / 促销，还是全部？
- `待确认` 什么级别的内容需要即时提醒？什么内容只进日报？什么内容只归档？
- `待确认` 输出语言是中文、日文、英文，还是中日双语？
- `待确认` 老板 dashboard 最关心 3 个指标是什么？
- `待确认` 后续是否需要接邮件、Slack、飞书、微信，还是先只保留网页和 API？

## 2. 背景

市场部需要持续关注目标公司、竞品和行业热点，但人工方式存在几个问题：

- 信息源分散：官网、PR、Google News、行业媒体、活动页分散在不同入口。
- 噪音很高：公司名出现在金融新闻、旧文章、目录页或弱相关内容中，并不等于值得推送。
- 老板需要总览：管理层关心“今天有没有值得看的信号”，不是逐条看搜索结果。
- 市场部需要动作：市场部关心这条动态能不能转化为 Newsletter、LinkedIn、展会材料、销售话术或竞品观察。

AIhot 的实现形式可作为参考：抓取公开动态，经过 AI / 规则筛噪后，形成精选流、日报、主题分类和 API / Agent 接入口。AI Hot Tracker 在此基础上做企业市场部场景改造。

## 3. 产品定位

AI Hot Tracker 不是普通新闻聚合站，也不是实时舆情系统。

它是一个面向目标公司与竞品的市场情报工作台：

- 每日从低成本公开来源收集候选动态。
- 通过规则和后续 AI 判断筛掉噪音。
- 将内容分为即时提醒、进入日报、归档观察。
- 给市场部提供选题池。
- 给老板提供可快速判断的 dashboard。

`待确认` 这里需要向需求方确认：他们是否接受“目标公司情报工作台”这个定义。如果他们实际想要的是公开新闻门户、内容运营选题库或竞品数据库，产品边界需要调整。

## 4. 用户与场景

### 4.1 目标用户

| 用户 | 主要诉求 | 使用频率 |
| --- | --- | --- |
| 市场部成员 | 找 Newsletter / LinkedIn / 活动素材选题，观察竞品动作 | 每日 |
| 市场部负责人 | 判断哪些动态值得分配给团队跟进 | 每日 / 每周 |
| 老板 / 管理层 | 快速查看目标公司和行业热点是否有重要变化 | 每日 / 按需 |
| 实习生 / 运营支持 | 维护公司名单、关键词、来源配置，人工复核结果 | 每日 / 每周 |

`待确认` 谁是第一优先用户：市场部执行者，市场部负责人，还是老板。如果第一优先用户不同，首页信息密度和默认视图会不同。

### 4.2 核心场景

1. 市场部每天打开 dashboard，查看精选信号。
2. 老板查看总览指标，判断今天是否有重要动态。
3. 团队通过公司筛选查看某一家公司的新闻。
4. 团队通过全部动态查看被归档的候选，检查是否有误杀。
5. 后续系统通过 API / JSON 给其他 Agent 或内部系统读取。

## 6. 产品目标

### 6.1 MVP 目标

验证“目标公司热点追踪”是否能以低成本方式帮助市场部：

- 每天快速知道目标公司和竞品是否有值得关注的动态。
- 减少人工搜索和噪音判断成本。
- 为老板提供一个可浏览的 dashboard。
- 为后续批量接入公司名单验证信息架构。

`当前假设` MVP 的核心验证指标不是“抓到越多新闻越好”，而是“默认精选是否足够少、足够相关、能辅助市场部行动”。

### 6.2 下一阶段目标

在 MVP 基础上增强到更接近 AIhot 的能力：

- 扩展公司池和竞品池。
- 增加主题页：ADC、CGT、GMP、CDMO、Japan Market 等。
- 引入 AI 摘要和业务动作建议。
- 增加人工反馈机制。
- 增加订阅 / 推送机制。
- 增加后台配置能力。

## 7. 功能需求

### 7.1 公司池管理

#### 当前

通过 `config/companies.json` 手动维护公司：

- 公司 ID
- 展示名称
- 别名
- 市场区域
- 战略主题关键词
- 业务动作关键词
- 噪音词

#### 下一阶段

需要后台界面或表格导入能力：

- 批量导入公司名单。
- 为每家公司配置关键词。
- 设置公司优先级：高 / 中 / 低。
- 设置公司类型：自家公司、竞品、客户、合作伙伴、行业标杆。

`待确认` 休假前公司名单需要补充到这里。需要确认名单中的公司是否都属于同一类对象，还是应分为竞品、客户、合作伙伴、行业标杆、潜在客户。

### 7.2 来源管理

#### 当前

通过 `config/sources.json` 手动维护来源：

- 官网新闻页
- 官方 RSS
- Google News RSS
- PR / 行业媒体间接来源

数据源完整盘点不直接写入运行配置，而是先维护公司级数据源档案：

- `docs/data_source_profile_template.md`

该档案用于记录完整情报地图，包括官网、RSS、新闻稿平台、行业媒体、LinkedIn、微信公众号、视频平台、展会、招聘、专利、论文、数据库、登录平台和反爬严重网站。是否进入自动抓取，由每个来源的 `enable_status` 单独决定。

#### 下一阶段

需要支持：

- 来源可用性检测。
- 403 / 404 / 超时提醒。
- 每个来源的可信度设置。
- 每个来源的抓取频率设置。
- 来源归属：公司自有、聚合器、行业媒体、PR wire。

### 7.3 候选收集

系统应每天定时抓取配置来源，并生成候选列表。

基础字段：

- 标题
- URL
- 来源
- 发布时间
- 公司
- 摘要
- 原始匹配关键词
- 抓取时间

验收标准：

- 每天定时任务可自动运行。
- 单个来源失败不阻断整体生成。
- 抓取错误要在报告中显示。

### 7.4 去重

系统需要对 URL 做标准化，避免同一条新闻重复出现。

规则包括：

- 去除 UTM 参数。
- 标准化域名大小写。
- 去除 URL fragment。
- 对标准化 URL 生成唯一 ID。

下一阶段可增加：

- 标题相似度去重。
- Google News 和原始媒体链接合并。
- 同一新闻多媒体转载归并。

### 7.5 相关性评分

当前评分规则：

- 公司别名命中加分。
- 公司自有来源加分。
- 战略主题命中加分。
- 业务动作命中加分。
- 噪音词命中扣分。
- 超过时效窗口降权。

下一阶段需要引入 AI 判断：

- 判断是否真正与公司业务相关。
- 判断是否属于竞品动态。
- 判断对日本市场是否有意义。
- 判断是否适合市场部内容转化。
- 输出推荐理由。

`待确认` 评分的“重要”到底由谁定义：老板、市场部负责人、内容运营，还是销售 / BD。不同角色对重要性的定义不同。

### 7.6 推送分层

内容分为三层：

| 分层 | 含义 | 当前动作 | 后续动作 |
| --- | --- | --- | --- |
| immediate | 高价值重大信号 | dashboard 标记 | 邮件 / Slack / 飞书 / 微信提醒 |
| daily | 值得每日阅读 | 默认展示在精选流 | 进入日报 |
| archive | 低价值或噪音 | 归档可查 | 用于复核和模型优化 |

`待确认` 哪些事件类型必须进入 `immediate`。建议在沟通时让需求方直接给 5 个“必须立刻提醒”的例子和 5 个“不应打扰”的例子。

## 8. 更新频率

### 当前频率

当前建议并已配置为工作日每天自动更新一次。

GitHub Actions 当前计划：

```text
0 0 * * 1-5
```

即 UTC 00:00，换算为日本时间约 09:00。

`当前假设` 日本时间上午 09:00 更适合工作流，因为市场部可以在当天安排内容或跟进动作。

### 频率原则

MVP 阶段不建议实时更新，原因：

- 市场部需要的是可读信息，不是秒级提醒。
- 免费 RSS / Google News 不保证实时性。
- 高频抓取会增加维护和风控成本。
- 低质量实时推送会降低信任度。

### 后续频率设计

- 普通日报：工作日每天 1 次。
- 高优先级公司：每天 2 次。
- 重大信号：每 1-3 小时扫描一次。
- 即时提醒：只对重大合作、新品、监管、竞品进入日本市场等高价值类别开放。

`待确认` 需求方是否真的需要“实时”。如果只是想每天知道行业动态，日报即可；如果担心错过竞品重大动作，才需要额外即时提醒。

## 9. 非功能需求

### 9.1 成本

MVP 必须优先使用免费公开来源：

- 官方 RSS
- 官网页面
- Google News RSS
- 静态 GitHub Pages
- GitHub Actions

除非确认业务价值，否则不接付费新闻 API。

### 9.2 可维护性

- 公司配置和来源配置必须结构化。
- 单个来源失败不能影响全局。
- 页面和数据分离。
- 静态 API 可被其他工具读取。

### 9.3 可解释性

每条内容必须提供：

- 分数
- 分层
- 分类
- 推荐 / 归档理由

避免黑箱式“AI 说重要”。

### 9.4 安全与合规

- 不抓取需要登录的内容。
- 不自动抓 LinkedIn 等高风险平台。
- 不绕过反爬或付费墙。
- 公开部署前避免包含内部公司名单、客户名单或未公开资料。

## 10. 成功指标

### MVP 指标

- 每日自动更新成功率 >= 90%。
- 单日候选抓取数 > 0。
- 默认精选数量控制在 3-20 条。当前样本规则下为 2 条，偏严格，后续可根据反馈放宽。
- dashboard 可被非技术用户打开并理解。
- 市场部能从精选中找到至少 1 条可跟进内容。

### 下一阶段指标

- 接入公司数 >= 20。
- 人工复核后误推率 < 20%。
- 关键竞品重要动态漏报率持续下降。
- 每周至少产出 3 条可用于市场内容或竞品分析的信号。

## 12. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 免费来源不稳定 | 抓取缺失或延迟 | 多来源冗余，错误展示但不阻断 |
| 噪音过多 | 用户失去信任 | 默认精选收紧，保留全量候选用于复核 |
| 重要新闻漏掉 | 影响业务判断 | 增加公司关键词、竞品关键词、人工反馈 |
| 公开部署泄露内部名单 | 合规风险 | 公开版只放样本公司，内部版另行部署 |
| AI 总结误导 | 判断错误 | 保留来源链接和规则理由，AI 只做辅助 |

## 13. 待确认问题

这一节是下一次和需求方交流时建议逐条问的问题。建议不要一次性问太散，可以按“目标、内容、推送、交付、权限”五组来问。

### 13.1 目标定义

1. `待确认` 你说的“热点新闻平台”，第一优先目标是什么：市场选题、竞品监控、销售线索、老板总览，还是品牌 / 行业观察？
2. `待确认` 这个工具是给市场部日常用，还是给老板看结果？两者的首页可能不同。
3. `待确认` 需求方希望它最终像 AIhot 一样偏资讯产品，还是偏企业内部情报系统？

### 13.2 公司和主题范围

4. `待确认` 休假前公司名单是否可以公开部署，还是必须做内部版？
5. `待确认` 哪些公司属于高优先级，需要更高频率扫描？
6. `待确认` 公司类型如何划分：竞品、客户、合作伙伴、供应商、行业标杆？
7. `待确认` 哪些主题是第一阶段重点：ADC、CGT、GMP、CDMO、日本市场，还是其他？

### 13.3 内容价值判断

8. `待确认` 什么内容算“值得推送”：新品、合作、融资、展会、临床 / 监管、白皮书、招聘、价格变化？
9. `待确认` 哪些内容明确不需要推送：股价、财报电话会、招聘、目录页、旧新闻、转载？
10. `待确认` 是否需要区分“公司自身新闻”和“公司所在领域热点”？
11. `待确认` 竞品动态是否需要和 ACRO 产品线建立映射，例如 ADC、CGT、GMP、蛋白、抗体、CDMO？

### 13.4 更新和推送

### 13.5 输出形式和权限
