# PRD: AI Hot Tracker 目标公司情报 Dashboard

## 1. 文档信息

- 产品名称：AI Hot Tracker
- 当前版本：MVP / GitHub Pages 可访问版本
- 文档日期：2026-07-07
- 主要用户：ACRO / 百普赛斯市场部、管理层
- 当前线上地址：https://tenltrist.github.io/acro-ai-hot-tracker/web/
- 当前样本公司：ACROBiosystems / 百普赛斯、Thermo Fisher Scientific

## 2. 背景

市场部需要持续关注目标公司、竞品和行业热点，但人工方式存在几个问题：

- 信息源分散：官网、PR、Google News、行业媒体、活动页分散在不同入口。
- 噪音很高：公司名出现在金融新闻、旧文章、目录页或弱相关内容中，并不等于值得推送。
- 老板需要总览：管理层关心“今天有没有值得看的信号”，不是逐条看搜索结果。
- 市场部需要动作：市场部关心这条动态能不能转化为 Newsletter、LinkedIn、展会材料、销售话术或竞品观察。

AIhot 的实现形式可作为参考：抓取公开动态，经过 AI / 规则筛噪后，形成精选流、日报、主题分类和 API / Agent 接入口。AI Hot Tracker 在此基础上做企业市场部场景改造。

## 3. 产品定位

AI Hot Tracker 不是普通新闻聚合站，也不是实时舆情系统。

它是一个面向目标公司与竞品的市场情报工作台：

- 每日从低成本公开来源收集候选动态。
- 通过规则和后续 AI 判断筛掉噪音。
- 将内容分为即时提醒、进入日报、归档观察。
- 给市场部提供选题池。
- 给老板提供可快速判断的 dashboard。

## 4. 用户与场景

### 4.1 目标用户

| 用户 | 主要诉求 | 使用频率 |
| --- | --- | --- |
| 市场部成员 | 找 Newsletter / LinkedIn / 活动素材选题，观察竞品动作 | 每日 |
| 市场部负责人 | 判断哪些动态值得分配给团队跟进 | 每日 / 每周 |
| 老板 / 管理层 | 快速查看目标公司和行业热点是否有重要变化 | 每日 / 按需 |
| 实习生 / 运营支持 | 维护公司名单、关键词、来源配置，人工复核结果 | 每日 / 每周 |

### 4.2 核心场景

1. 市场部每天打开 dashboard，查看精选信号。
2. 老板查看总览指标，判断今天是否有重要动态。
3. 团队通过公司筛选查看某一家公司的新闻。
4. 团队通过全部动态查看被归档的候选，检查是否有误杀。
5. 后续系统通过 API / JSON 给其他 Agent 或内部系统读取。

## 5. 当前 MVP 状态

### 5.1 已实现能力

- GitHub Pages 线上 dashboard。
- 公司池：ACROBiosystems / 百普赛斯、Thermo Fisher Scientific。
- 公开来源抓取：官网 / 官方 RSS / Google News RSS。
- 候选去重。
- 规则评分。
- 推送分层：`immediate`、`daily`、`archive`。
- Dashboard 指标：新候选、建议推送、即时提醒、噪音压制。
- 筛选器：按分层、公司、主题筛选。
- 主题热度和来源构成。
- 静态 API：
  - `/api/public/daily.json`
  - `/api/public/items.json`
  - `/api/public/topics.json`
- 单文件分享版 HTML。
- GitHub Actions 自动部署。
- 工作日每日自动更新任务。

### 5.2 当前数据表现

当前最近一次生成结果：

- 新候选：203 条
- 进入日报：2 条
- 即时提醒：0 条
- 归档观察：201 条
- 公司数量：2 家
- 来源数量：6 个
- 已知抓取警告：Thermo Fisher newsroom HTML 页面返回 403，但官方 RSS 和 Google News 来源可用。

### 5.3 当前不能解决的问题

- 还不是完整 SaaS，没有登录、权限、后台管理。
- 还没有真正的 LLM 深度总结和业务建议。
- 还没有人工反馈闭环，无法自动学习“误推 / 漏推”。
- 还没有完整竞品库和行业主题库。
- 即时提醒只是分层标记，还没有接邮件、Slack、微信或飞书。
- 公开 RSS / Google News 有覆盖不稳定和时效不一致的问题。

## 6. 产品目标

### 6.1 MVP 目标

验证“目标公司热点追踪”是否能以低成本方式帮助市场部：

- 每天快速知道目标公司和竞品是否有值得关注的动态。
- 减少人工搜索和噪音判断成本。
- 为老板提供一个可浏览的 dashboard。
- 为后续批量接入公司名单验证信息架构。

### 6.2 下一阶段目标

在 MVP 基础上增强到更接近 AIhot 的能力：

- 扩展公司池和竞品池。
- 增加主题页：ADC、CGT、GMP、CDMO、Japan Market 等。
- 引入 AI 摘要和业务动作建议。
- 增加人工反馈机制。
- 增加订阅 / 推送机制。
- 增加后台配置能力。

## 7. 功能需求

### 7.1 公司池管理

#### 当前

通过 `config/companies.json` 手动维护公司：

- 公司 ID
- 展示名称
- 别名
- 市场区域
- 战略主题关键词
- 业务动作关键词
- 噪音词

#### 下一阶段

需要后台界面或表格导入能力：

- 批量导入公司名单。
- 为每家公司配置关键词。
- 设置公司优先级：高 / 中 / 低。
- 设置公司类型：自家公司、竞品、客户、合作伙伴、行业标杆。

### 7.2 来源管理

#### 当前

通过 `config/sources.json` 手动维护来源：

- 官网新闻页
- 官方 RSS
- Google News RSS
- PR / 行业媒体间接来源

#### 下一阶段

需要支持：

- 来源可用性检测。
- 403 / 404 / 超时提醒。
- 每个来源的可信度设置。
- 每个来源的抓取频率设置。
- 来源归属：公司自有、聚合器、行业媒体、PR wire。

### 7.3 候选收集

系统应每天定时抓取配置来源，并生成候选列表。

基础字段：

- 标题
- URL
- 来源
- 发布时间
- 公司
- 摘要
- 原始匹配关键词
- 抓取时间

验收标准：

- 每天定时任务可自动运行。
- 单个来源失败不阻断整体生成。
- 抓取错误要在报告中显示。

### 7.4 去重

系统需要对 URL 做标准化，避免同一条新闻重复出现。

规则包括：

- 去除 UTM 参数。
- 标准化域名大小写。
- 去除 URL fragment。
- 对标准化 URL 生成唯一 ID。

下一阶段可增加：

- 标题相似度去重。
- Google News 和原始媒体链接合并。
- 同一新闻多媒体转载归并。

### 7.5 相关性评分

当前评分规则：

- 公司别名命中加分。
- 公司自有来源加分。
- 战略主题命中加分。
- 业务动作命中加分。
- 噪音词命中扣分。
- 超过时效窗口降权。

下一阶段需要引入 AI 判断：

- 判断是否真正与公司业务相关。
- 判断是否属于竞品动态。
- 判断对日本市场是否有意义。
- 判断是否适合市场部内容转化。
- 输出推荐理由。

### 7.6 推送分层

内容分为三层：

| 分层 | 含义 | 当前动作 | 后续动作 |
| --- | --- | --- | --- |
| immediate | 高价值重大信号 | dashboard 标记 | 邮件 / Slack / 飞书 / 微信提醒 |
| daily | 值得每日阅读 | 默认展示在精选流 | 进入日报 |
| archive | 低价值或噪音 | 归档可查 | 用于复核和模型优化 |

### 7.7 Dashboard

Dashboard 应包含：

- 老板总览指标。
- 精选信号流。
- 全部动态筛选。
- 公司筛选。
- 主题筛选。
- 主题热度。
- 来源构成。
- AIhot 式 API 入口。
- 市场部动作队列。

当前已实现这些模块。

下一阶段可增加：

- 趋势图。
- 公司矩阵。
- 竞品对比。
- 主题详情页。
- 日报归档页。
- 人工反馈按钮：有用 / 无用 / 误判 / 需要跟进。

### 7.8 日报

当前日报以 Markdown 和 JSON 两种形式输出。

日报应包含：

- 日期和生成时间。
- 总览指标。
- 即时提醒候选。
- 今日简报。
- 归档观察。
- 抓取错误。
- 下一步确认问题。

下一阶段日报可增加：

- 中文业务摘要。
- 给市场部的建议动作。
- 给老板的 3 行 summary。
- 按公司和主题分组。
- 邮件发送。

### 7.9 静态 API / Agent 接入

当前已提供：

- `daily.json`：日报精选数据。
- `items.json`：全量候选。
- `topics.json`：主题、来源、公司元数据。

下一阶段可增加：

- `companies.json`：公司画像。
- `signals.json`：只含高价值信号。
- `archive.json`：归档数据。
- RSS 输出。
- Agent Skill 接入说明。

## 8. 更新频率

### 当前频率

当前建议并已配置为工作日每天自动更新一次。

GitHub Actions 当前计划：

```text
0 0 * * 1-5
```

即 UTC 00:00，换算为日本时间约 09:00。

### 频率原则

MVP 阶段不建议实时更新，原因：

- 市场部需要的是可读信息，不是秒级提醒。
- 免费 RSS / Google News 不保证实时性。
- 高频抓取会增加维护和风控成本。
- 低质量实时推送会降低信任度。

### 后续频率设计

- 普通日报：工作日每天 1 次。
- 高优先级公司：每天 2 次。
- 重大信号：每 1-3 小时扫描一次。
- 即时提醒：只对重大合作、新品、监管、竞品进入日本市场等高价值类别开放。

## 9. 非功能需求

### 9.1 成本

MVP 必须优先使用免费公开来源：

- 官方 RSS
- 官网页面
- Google News RSS
- 静态 GitHub Pages
- GitHub Actions

除非确认业务价值，否则不接付费新闻 API。

### 9.2 可维护性

- 公司配置和来源配置必须结构化。
- 单个来源失败不能影响全局。
- 页面和数据分离。
- 静态 API 可被其他工具读取。

### 9.3 可解释性

每条内容必须提供：

- 分数
- 分层
- 分类
- 推荐 / 归档理由

避免黑箱式“AI 说重要”。

### 9.4 安全与合规

- 不抓取需要登录的内容。
- 不自动抓 LinkedIn 等高风险平台。
- 不绕过反爬或付费墙。
- 公开部署前避免包含内部公司名单、客户名单或未公开资料。

## 10. 成功指标

### MVP 指标

- 每日自动更新成功率 >= 90%。
- 单日候选抓取数 > 0。
- 默认精选数量控制在 3-20 条。
- dashboard 可被非技术用户打开并理解。
- 市场部能从精选中找到至少 1 条可跟进内容。

### 下一阶段指标

- 接入公司数 >= 20。
- 人工复核后误推率 < 20%。
- 关键竞品重要动态漏报率持续下降。
- 每周至少产出 3 条可用于市场内容或竞品分析的信号。

## 12. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 免费来源不稳定 | 抓取缺失或延迟 | 多来源冗余，错误展示但不阻断 |
| 噪音过多 | 用户失去信任 | 默认精选收紧，保留全量候选用于复核 |
| 重要新闻漏掉 | 影响业务判断 | 增加公司关键词、竞品关键词、人工反馈 |
| 公开部署泄露内部名单 | 合规风险 | 公开版只放样本公司，内部版另行部署 |
| AI 总结误导 | 判断错误 | 保留来源链接和规则理由，AI 只做辅助 |
